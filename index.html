üèóÔ∏è Creating Tables (DDL)

-- Creating a Students table CREATE TABLE students ( student_id INT PRIMARY KEY AUTO_INCREMENT, first_name VARCHAR(50) NOT NULL, last_name VARCHAR(50) NOT NULL, email VARCHAR(100) UNIQUE, age INT CHECK (age >= 18), enrollment_date DATE DEFAULT CURRENT_DATE, gpa DECIMAL(3,2), major VARCHAR(50) ); -- Creating a Courses table CREATE TABLE courses ( course_id INT PRIMARY KEY AUTO_INCREMENT, course_name VARCHAR(100) NOT NULL, course_code VARCHAR(10) UNIQUE, credits INT DEFAULT 3, instructor VARCHAR(50), department VARCHAR(50) ); -- Creating a junction table for many-to-many relationship CREATE TABLE student_courses ( enrollment_id INT PRIMARY KEY AUTO_INCREMENT, student_id INT, course_id INT, enrollment_date DATE DEFAULT CURRENT_DATE, grade CHAR(2), FOREIGN KEY (student_id) REFERENCES students(student_id), FOREIGN KEY (course_id) REFERENCES courses(course_id) );

üîç Querying Data (DQL)

-- Basic SELECT queries SELECT * FROM students; SELECT first_name, last_name, gpa FROM students WHERE gpa > 3.5; SELECT * FROM students WHERE major = 'Computer Science' ORDER BY gpa DESC; -- Aggregate functions SELECT COUNT(*) as total_students FROM students; SELECT major, AVG(gpa) as average_gpa FROM students GROUP BY major; SELECT major, COUNT(*) as student_count FROM students GROUP BY major HAVING COUNT(*) > 1; -- JOINS - Most Important for Interviews! -- Inner Join: Students with their enrolled courses SELECT s.first_name, s.last_name, c.course_name, sc.grade FROM students s INNER JOIN student_courses sc ON s.student_id = sc.student_id INNER JOIN courses c ON sc.course_id = c.course_id; -- Left Join: All students (even those not enrolled) SELECT s.first_name, s.last_name, c.course_name FROM students s LEFT JOIN student_courses sc ON s.student_id = sc.student_id LEFT JOIN courses c ON sc.course_id = c.course_id; -- Complex query with subquery SELECT first_name, last_name, gpa FROM students WHERE gpa > (SELECT AVG(gpa) FROM students); -- Window functions (Advanced) SELECT first_name, last_name, gpa, RANK() OVER (ORDER BY gpa DESC) as gpa_rank, ROW_NUMBER() OVER (PARTITION BY major ORDER BY gpa DESC) as major_rank FROM students;

// Update operations db.students.updateOne( {"email": "john.doe@email.com"}, {"$set": {"gpa": 3.80, "year": "Sophomore"}} ) db.students.updateMany( {"major": "Computer Science"}, {"$push": {"skills": "JavaScript"}} ) // Delete operations db.students.deleteOne({"email": "john.doe@email.com"}) db.students.deleteMany({"gpa": {"$lt": 3.0}}) // Aggregation Pipeline - Very Important for Interviews! db.students.aggregate([ // Stage 1: Match students with GPA > 3.5 { "$match": { "gpa": {"$gt": 3.5} } }, // Stage 2: Group by major and calculate average GPA { "$group": { "_id": "$major", "averageGPA": {"$avg": "$gpa"}, "studentCount": {"$sum": 1}, "topStudent": {"$max": "$gpa"} } }, // Stage 3: Sort by average GPA descending { "$sort": { "averageGPA": -1 } } ]) // Complex aggregation with lookup (like JOIN) db.students.aggregate([ { "$lookup": { "from": "courses", "localField": "courses.courseId", "foreignField": "_id", "as": "courseDetails" } }, { "$project": { "firstName": 1, "lastName": 1, "major": 1, "courseCount": {"$size": "$courses"}, "courseNames": "$courseDetails.courseName" } } ]) // Text search db.students.createIndex({"firstName": "text", "lastName": "text"}) db.students.find({"$text": {"$search": "John"}}) // Geospatial queries (if you have location data) db.students.find({ "location": { "$near": { "$geometry": {"type": "Point", "coordinates": [-71.0589, 42.3601]}, "$maxDistance": 1000 } } })

üìè Normalization Forms

First Normal Form (1NF)

Rule: Each column should contain atomic (indivisible) values, and each column should contain values of a single type.

‚ùå Violating 1NF:

-- BAD: Multiple values in a single column CREATE TABLE students_bad ( student_id INT, name VARCHAR(100), subjects VARCHAR(200) -- "Math, Physics, Chemistry" - violates 1NF );

‚úÖ Following 1NF:

-- GOOD: Atomic values CREATE TABLE students ( student_id INT, name VARCHAR(100) ); CREATE TABLE student_subjects ( student_id INT, subject VARCHAR(50) );

Second Normal Form (2NF)

Rule: Must be in 1NF and all non-key attributes must be fully functionally dependent on the primary key.

‚ùå Violating 2NF:

-- BAD: Partial dependency CREATE TABLE student_courses_bad ( student_id INT, course_id INT, student_name VARCHAR(100), -- Depends only on student_id course_name VARCHAR(100), -- Depends only on course_id grade CHAR(2), PRIMARY KEY (student_id, course_id) );

‚úÖ Following 2NF:

-- GOOD: Separate tables CREATE TABLE students ( student_id INT PRIMARY KEY, student_name VARCHAR(100) ); CREATE TABLE courses ( course_id INT PRIMARY KEY, course_name VARCHAR(100) ); CREATE TABLE enrollments ( student_id INT, course_id INT, grade CHAR(2), PRIMARY KEY (student_id, course_id) );

Third Normal Form (3NF)

Rule: Must be in 2NF and have no transitive dependencies (non-key attributes should not depend on other non-key attributes).

‚ùå Violating 3NF:

-- BAD: Transitive dependency CREATE TABLE students_bad ( student_id INT PRIMARY KEY, name VARCHAR(100), department_id INT, department_name VARCHAR(100), -- Depends on department_id, not student_id department_head VARCHAR(100) -- Depends on department_id, not student_id );

‚úÖ Following 3NF:

-- GOOD: Remove transitive dependencies CREATE TABLE departments ( department_id INT PRIMARY KEY, department_name VARCHAR(100), department_head VARCHAR(100) ); CREATE TABLE students ( student_id INT PRIMARY KEY, name VARCHAR(100), department_id INT, FOREIGN KEY (department_id) REFERENCES departments(department_id) );

Boyce-Codd Normal Form (BCNF)

Rule: Must be in 3NF and for every functional dependency X ‚Üí Y, X must be a superkey.

‚ùå Violating BCNF:

-- BAD: Non-superkey determines another attribute CREATE TABLE course_instructor_bad ( course_id INT, instructor_id INT, instructor_office VARCHAR(50), -- instructor_id ‚Üí instructor_office PRIMARY KEY (course_id, instructor_id) -- Problem: instructor_id is not a superkey but determines instructor_office );

‚úÖ Following BCNF:

-- GOOD: Separate the dependency CREATE TABLE instructors ( instructor_id INT PRIMARY KEY, instructor_office VARCHAR(50) ); CREATE TABLE course_instructor ( course_id INT, instructor_id INT, PRIMARY KEY (course_id, instructor_id), FOREIGN KEY (instructor_id) REFERENCES instructors(instructor_id) );
üîê ACID Properties & Transactions
üõ°Ô∏è ACID Properties
ACID ensures database transactions are processed reliably and maintain data integrity.

üîç ACID Explained

‚öõÔ∏è ATOMICITY
"All or Nothing"
Either all operations succeed or all fail
Example: Money transfer - debit and credit both happen or neither
üîí CONSISTENCY
"Valid State"
Database remains in valid state before/after transaction
Example: Account balance can't be negative
üèùÔ∏è ISOLATION
"Concurrent Safety"
Transactions don't interfere with each other
Example: Two people can't book the same seat simultaneously
üíæ DURABILITY
"Permanent Storage"
Committed changes persist even after system failure
Example: Confirmed payment survives server crash

üí≥ Banking Transaction Example

-- Bank transfer transaction demonstrating ACID START TRANSACTION; -- Check if source account has sufficient balance SELECT balance FROM accounts WHERE account_id = 1001 FOR UPDATE; -- If balance >= transfer_amount, proceed UPDATE accounts SET balance = balance - 500 WHERE account_id = 1001; UPDATE accounts SET balance = balance + 500 WHERE account_id = 1002; -- Log the transaction INSERT INTO transaction_log (from_account, to_account, amount, timestamp) VALUES (1001, 1002, 500, NOW()); -- If all operations successful COMMIT; -- If any operation fails -- ROLLBACK;

üìö Indexing & Performance
üöÄ What is Indexing?
Indexes are data structures that improve query performance by creating shortcuts to find data quickly, like an index in a book.

üìñ Index Analogy

üìö Without Index

Finding "Python" in a book:
‚Ä¢ Read page 1, 2, 3... 500
‚Ä¢ Linear search through all pages
‚Ä¢ Time: O(n) - Very slow!

Like scanning entire table
üìë With Index

Finding "Python" in a book:
‚Ä¢ Check index: "Python - Page 287"
‚Ä¢ Jump directly to page 287
‚Ä¢ Time: O(log n) - Very fast!

Like using B-tree index

üóÇÔ∏è Types of Indexes

üîë PRIMARY INDEX
‚Ä¢ Automatically created for primary key
‚Ä¢ Unique and clustered
‚Ä¢ Data pages physically sorted
üéØ UNIQUE INDEX
‚Ä¢ Ensures uniqueness
‚Ä¢ Automatically created for UNIQUE constraints
‚Ä¢ Can have multiple per table
üìä COMPOSITE INDEX
‚Ä¢ Multiple columns together
‚Ä¢ Order matters: (A, B) ‚â† (B, A)
‚Ä¢ Useful for multi-column WHERE clauses
üîç PARTIAL INDEX
‚Ä¢ Index on subset of data
‚Ä¢ WHERE condition in index
‚Ä¢ Saves space and improves performance

üõ†Ô∏è Creating Indexes in SQL

-- Basic index on single column CREATE INDEX idx_student_email ON students(email); -- Composite index on multiple columns CREATE INDEX idx_student_major_gpa ON students(major, gpa); -- Unique index CREATE UNIQUE INDEX idx_student_student_id ON students(student_id); -- Partial index (PostgreSQL) CREATE INDEX idx_active_students ON students(last_name) WHERE is_active = true; -- Covering index (includes additional columns) CREATE INDEX idx_student_lookup ON students(student_id) INCLUDE (first_name, last_name, email); -- Text search index CREATE INDEX idx_student_name_search ON students USING gin(to_tsvector('english', first_name || ' ' || last_name)); -- Analyzing index usage EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM students WHERE email = 'john.doe@email.com'; -- Show all indexes on a table SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'students';

‚öñÔ∏è CAP Theorem

CAP Theorem: In a distributed system, you can only guarantee 2 out of 3 properties: Consistency, Availability, and Partition tolerance.

üîí CONSISTENCY
All nodes see the same data simultaneously

Example: Banking transactions must be consistent across all servers
üåê AVAILABILITY
System remains operational

Example: Website should always respond, even if some servers are down
üîå PARTITION TOLERANCE
System continues despite network failures

Example: System works even when servers can't communicate

üéØ CAP Trade-offs in Real Systems

System Choice Trade-off Use Case
MySQL CA (Consistency + Availability) Not partition tolerant Single datacenter applications
MongoDB CP (Consistency + Partition tolerance) May become unavailable Strong consistency requirements
Cassandra AP (Availability + Partition tolerance) Eventually consistent High availability systems
Redis CP (with clustering) May lose availability Caching with consistency

Q2: Explain the difference between INNER JOIN and LEFT JOIN with an example.

-- Sample data Students: [John(1), Jane(2), Bob(3)] Enrollments: [John‚ÜíCS101, Jane‚ÜíMATH201] -- INNER JOIN: Only students who are enrolled SELECT s.name, e.course FROM students s INNER JOIN enrollments e ON s.id = e.student_id; -- Result: John‚ÜíCS101, Jane‚ÜíMATH201 (Bob excluded) -- LEFT JOIN: All students, with or without enrollments SELECT s.name, e.course FROM students s LEFT JOIN enrollments e ON s.id = e.student_id; -- Result: John‚ÜíCS101, Jane‚ÜíMATH201, Bob‚ÜíNULL
Q3: What is the N+1 query problem?

Problem: Making N additional queries for N records

-- BAD: N+1 Problem -- 1 query to get students SELECT * FROM students; -- Returns 100 students -- Then N queries (one for each student) SELECT * FROM courses WHERE student_id = 1; SELECT * FROM courses WHERE student_id = 2; -- ... 100 more queries! -- GOOD: Single query with JOIN SELECT s.*, c.course_name FROM students s LEFT JOIN student_courses sc ON s.id = sc.student_id LEFT JOIN courses c ON sc.course_id = c.id;
Q4: How would you design a database for a social media platform?

Approach:
1. Identify entities: Users, Posts, Comments, Likes, Follows
2. Choose database: Hybrid approach (SQL + NoSQL)
3. User data: SQL for consistency (profiles, auth)
4. Posts/Feed: NoSQL for scalability (MongoDB/Cassandra)
5. Real-time features: Redis for caching, notifications
6. Analytics: Data warehouse (BigQuery, Snowflake)
